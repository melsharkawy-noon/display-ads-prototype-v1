1. PROJECT OVERVIEW
================================================================================

1a. What This Is
────────────────
noon.com is one of the largest e-commerce platforms in the Middle East, operating
in the UAE, Saudi Arabia, and Egypt. Like any major e-commerce platform, noon
sells advertising space on its website — banner ads that appear on the homepage,
search results, product pages, category pages, cart, and checkout. These are
called "Display Ads."

This prototype is an interactive, high-fidelity mockup of the tool that would be
used to CREATE those display ad campaigns. Think of it as the "campaign builder"
— the interface where someone decides: what pages and ad slots to run on, who to
target, what budget to set, what the ad creative looks like, and when to run it.

It is NOT a production application. There is no backend, no API, no database.
Everything runs on mock data in the browser. The purpose is to explore and
validate the UI/UX before real engineering work begins — to get alignment with
product managers, designers, and stakeholders on how the campaign creation
experience should work.

1b. Who Uses It (The Two User Types)
─────────────────────────────────────
There are two very different users of this tool, and the interface adapts to each:

  SELF-SERVE CAMPAIGNS (code: entryType="seller")
    These are marketplace sellers — businesses that sell products on noon and want
    to promote them. They use a simplified, streamlined interface. They only get
    CPM (pay-per-impression) campaigns, basic targeting, and a straightforward
    product selection flow for their landing page. Think of a phone accessories
    seller wanting to boost visibility for their products.

  MANAGED CAMPAIGNS (code: entryType="brand")
    These are handled by noon's internal Operations team on behalf of large brands
    (Samsung, Nike, L'Oréal, etc.). The Ops team needs much more control: they
    can choose between CPM and CPT (pay-per-time-slot) pricing, use advanced
    compound audience targeting (AND multiple segments together), set per-zone
    bid multipliers, apply advertising credits, and get detailed budget/spend
    breakdowns. These campaigns are always budgeted in USD regardless of country.

1c. The Two Pricing Models
──────────────────────────
  CPM (Cost Per Mille / Cost Per Thousand Impressions):
    The advertiser sets a bid for every 1,000 times their ad is shown. They
    select which page zones to target (or exclude), set a daily or total budget,
    and the system serves their ads competitively based on their bid. This is the
    standard model for both self-serve and managed campaigns.

  CPT (Cost Per Time — managed campaigns only):
    The advertiser buys specific ad slots for specific time periods. Like renting
    a billboard — you pay a fixed rate for a slot (e.g., "Homepage Hero Banner 1")
    for specific hours or full days. There's a calendar/availability system that
    shows which slots are already booked. Ops teams typically reserve slots first
    (to hold them), then fill in the landing page and creative details later.

1d. The Campaign Creation Flow
──────────────────────────────
The user walks through a single scrolling page (not a step-by-step wizard) to
build a campaign. The sections appear progressively based on earlier choices.
The country selector lives in the left sidebar and is always available — it is
NOT a section step.

The two flows differ significantly:

  SELF-SERVE (seller) flow:
    1. Entry type is set (Self-Serve) via the header dropdown
    2. Schedule & Details — campaign name, start/end dates, budget, marketplace
    3. Landing Page — select products or provide a URL
    4. Targeting — simple audience segments (re-engagement, demographic)
    5. Creatives — upload banners or use the template builder
    6. Review + Submit

  MANAGED (brand) flow:
    1. Entry type is set (Managed) via the header dropdown
    2. Campaign Type — Endemic or Non-Endemic
       - Endemic = the brand sells products on noon (e.g., Samsung phones)
       - Non-Endemic = external advertiser, products NOT on noon (e.g., a car brand)
    3. Pricing Model — CPM or CPT
    4. (CPM) Schedule, budget, slot/zone selection OR (CPT) Slot selection + calendar booking
    5. Landing Page — select products or provide a URL
    6. (CPM only) Targeting — compound audience segments + slot targeting with multipliers
    7. (CPM only) Bidding — bid amount, frequency capping, budget pacing
    8. Creatives — upload banners or use the template builder
    9. Review + Submit (with estimated spend, credits, out-of-pocket)

  See sections 13 (scroll-spy), 16 (budget math), 17 (multipliers), 18 (compound
  targeting), 19 (CPT booking), 20 (landing page), 21 (creatives) in Part II
  for deep implementation details on each step.

1e. What the Prototype Demonstrates
────────────────────────────────────
  - The full end-to-end campaign creation experience for both user types
  - How pricing models (CPM vs CPT) change the available options
  - CPT slot booking with calendar availability and reservation
  - Advanced audience targeting with compound AND segments
  - Slot-level bid multipliers with recommendations
  - Budget transparency (estimated daily spend, credits, out-of-pocket)
  - AI-powered creative generation (mocked)
  - Product selection with search and filters for landing pages
  - A calendar overview showing booked slot inventory across time

1f. Technical Details
─────────────────────
Tech Stack:
  - Next.js 14 (App Router) with React 18, TypeScript
  - Tailwind CSS for styling
  - clsx + tailwind-merge via a cn() utility for conditional classnames
  - Lucide React for icons
  - date-fns for date utilities
  - No component library (shadcn-style custom primitives in src/components/ui/)

All data is mocked in src/lib/mock-data.ts. No network requests are made.

Run: npm run dev (port 3000)
Build: npm run build

2. NAMING CONVENTIONS & TERMINOLOGY
================================================================================

IMPORTANT: The codebase uses internal code names that differ from UI labels.
Always use these UI labels in the interface:

  Code Value          UI Display Label
  ─────────────────   ─────────────────────
  entryType="seller"  "Self-Serve Campaign"
  entryType="brand"   "Managed Campaign"
  campaignType="internal"    "Endemic"
  campaignType="third_party" "Non-Endemic"
  pricingModel="cpm"  "CPM" (Cost Per Mille)
  pricingModel="cpt"  "CPT" (Cost Per Time)

Domain Terms:
  - CLP: Category Landing Page (e.g., Electronics, Fashion, Beauty)
  - Page: A webpage on noon.com (Homepage, Search, PDP, CLPs, Cart, Checkout)
  - Slot: An individual ad placement on a page (e.g., "Hero Banner 1")
  - Zone: A group of related slots on a page (e.g., "Hero Banners" groups
    Hero Banner 1 + Hero Banner 2). Used for CPM grouping.
  - Endemic: Campaigns for brands/products sold ON noon (code: campaignType="internal")
  - Non-Endemic: Campaigns for brands/products NOT sold on noon (third-party)
  - Express: Fulfilled by noon (noon's warehouse/delivery)
  - Supermall: Marketplace sellers (third-party fulfillment)
  - noon: The e-commerce platform (always lowercase "noon")

Budget Notes:
  - Managed campaigns are ALWAYS in USD regardless of country
  - Self-serve campaigns use local currency (AED, SAR, EGP)

Time Conventions:
  - CPT booking days follow a 9 AM to 9 AM convention (not midnight to midnight)
  - Hourly CPT: Specific hours within a day
  - Daily CPT: Full 24-hour day booking

via a dropdown in the sticky header bar.

See section 1d for the per-flow step breakdown. The key section IDs used
internally are:
  Seller:    entry, seller-schedule, landing, seller-targeting, creatives
  Brand CPM: entry, campaign-type, pricing-model, schedule, landing, targeting, bidding, creatives
  Brand CPT: entry, campaign-type, pricing-model, cpt-calendar, landing, creatives

Conditional Section Visibility (detailed in section 13):
  - visibleSections is computed in SinglePageFlow.tsx based on draft state
  - CPT-specific sections only show for pricingModel="cpt"
  - CPM-specific sections only show for pricingModel="cpm"
  - Targeting and Bidding are only shown for CPM campaigns
  - Some sections differ significantly between seller and brand flows
  - Later sections only appear when earlier prerequisites are met (e.g., budget ≥ 100)

7. KEY DESIGN DECISIONS & RATIONALE
================================================================================

7a. Single-Page Wizard Over Step-By-Step
  The app originally had a step-by-step wizard (src/components/wizard/).
  It was replaced with SinglePageFlow.tsx — a single scrolling page with
  scroll-spy navigation. This gives better context (see all sections),
  faster navigation, and matches stakeholder preference.

7b. Section Extraction Refactor
  SinglePageFlow.tsx was originally 7,175 lines. It was decomposed into
  6 memoized section components to improve performance and maintainability.
  Each section uses React.memo to prevent unnecessary re-renders.

7c. Compound Audience Segments (Managed Flow Only)
  Self-serve: Simple segments (one condition = one segment).
  Managed: Can AND multiple conditions into compound segments.
  This was a product requirement to support advanced targeting for
  brands with dedicated Ops support.

7d. Slot Targeting Modes
  - "All Zones": Campaign runs on all available zones
  - "Include Specific": Pick zones, each page group gets its own multiplier
    with a recommendation based on daily views
  - "Exclude Specific": Pick zones to exclude, single global multiplier
  The per-group multiplier for include mode was a product requirement
  to give Ops more granular bidding control.

7e. AI Creative Generation
  The template builder supports AI generation of FULL ad banners.
  This is currently mock (2.5s delay, returns SVG placeholders).
  The AI takes logo + headlines + user prompt and produces complete
  banners, not just product images. The generated creative fills
  the entire ad space.

7f. Budget Display (Managed Flow)
  Budget summary shows: input budget, duration, derived daily/total spend,
  credits applied, out-of-pocket cost. Even for total budgets, the
  estimated daily spend is calculated and shown.

7g. Reserve Slots (CPT)
  Both hourly and daily CPT bookings have a "Reserve These Slots" button.
  This lets Ops hold inventory while they finish the landing page and
  creatives. Reservation is a local UI state (not persisted).

8. KNOWN ISSUES & PRE-EXISTING LINT ERRORS
================================================================================

These TypeScript errors exist across the codebase and were NOT introduced
by recent changes. They should be addressed in a future cleanup:

1. HTMLElement | null vs HTMLDivElement | null:
   Multiple refs in SinglePageFlow.tsx have this mismatch.
   Fix: Use HTMLElement type instead of HTMLDivElement, or cast.

2. boolean | null vs boolean | undefined:
   SinglePageFlow.tsx line ~1355. A null check mismatch.

3. StepBidding.tsx references "cpd" pricing model:
   Dead code referencing a pricing model that doesn't exist in PricingModel type.
   Also references slot.cpdRateUsd which doesn't exist on the Slot interface.

4. StepBidding.tsx / StepCountryPricing.tsx:
   Alert component receives className prop it doesn't support.

5. Legacy wizard/ components:
   These are NOT the primary flow. They may have stale types and props.
   The primary flow is SinglePageFlow.tsx + sections/*.tsx.

10. EXTENDING THE APPLICATION
================================================================================

Adding a New Section:
  1. Create src/components/sections/YourSection.tsx
  2. Wrap in React.memo, accept sectionRef prop
  3. Use useCampaign() for draft state access
  4. Import and render in SinglePageFlow.tsx
  5. Add to visibleSections logic if conditionally shown
  6. Add scroll-spy ID matching the sidebar navigation

Adding New Mock Data:
  1. Define interface in src/lib/types.ts
  2. Add data to src/lib/mock-data.ts
  3. Import where needed

Adding New Campaign Draft Fields:
  1. Add to CampaignDraft interface in types.ts
  2. Add default value to initialDraft in types.ts
  3. Use updateDraft({ newField: value }) in components

Common Patterns:
  - Conditional rendering based on draft.entryType, draft.campaignType,
    draft.pricingModel
  - Two-column layouts for selection panels (filters | items | selected)
  - Card-based sections with consistent header patterns
  - Toggle buttons for mode switches (not radio buttons)

13. SCROLL-SPY NAVIGATION (SinglePageFlow.tsx)
================================================================================
(Expands on section 3c above)

The left sidebar highlights which section the user is currently viewing.
This is powered by a scroll-spy system. (The header bar shows the entry type
dropdown and Save/Submit buttons — it does NOT display the active section.)

How it works:
  1. sectionRefs = useRef<Record<string, HTMLDivElement | null>>({})
     Each section component receives a ref callback: ref={el => sectionRefs.current["section-id"] = el}

  2. Three section lists exist depending on flow:
     - getCPMSections(): entry, campaign-type, pricing-model, schedule, landing, targeting, bidding, creatives
     - getCPTSections(): entry, campaign-type, pricing-model, cpt-calendar, landing, creatives
     - getSellerSections(): entry, seller-schedule, landing, seller-targeting, creatives
     SECTIONS = useMemo based on draft.entryType and draft.pricingModel

  3. visibleSections (useMemo) determines which sections are rendered:
     Seller flow: ["entry", "seller-schedule", then "landing"+"seller-targeting"+"creatives"
       if campaignName is set, startDate exists, and budget >= 100]
     Brand CPM: ["entry", "campaign-type", "pricing-model", "schedule", then
       "landing"+"targeting"+"bidding"+"creatives" if startDate exists and budget >= 100]
     Brand CPT: ["entry", "campaign-type", "pricing-model", then "cpt-calendar"
       if slotIds.length > 0, then "landing"+"creatives" if cptTotalHours > 0]

  4. Scroll event listener checks getBoundingClientRect().top for each visible
     section. The last one whose top is above headerOffset (150px) wins.

  5. scrollToSection() programmatically scrolls and sets isScrollingRef=true
     for 800ms to prevent the scroll spy from fighting the animation.

Sidebar navigation:
  - Each nav item shows an icon + label
  - Non-visible sections are hidden (opacity-0, h-0)
  - Active section: primary-500 bg, white text, font-semibold
  - Completed (above active): green dot indicator
  - Below sidebar: Country selector (always visible, changes draft.country)

15. ESTIMATED REACH CALCULATION
================================================================================

Location: SinglePageFlow.tsx, estimatedReach useMemo

  1. Determine relevant slots based on pricing model + targeting mode:
     - CPT: only selectedSlots (draft.slotIds)
     - CPM "all": all slots
     - CPM "include": slots matching draft.slotIds (or all if none selected)
     - CPM "exclude": all slots minus draft.excludedSlotIds

  2. Sum avgDailyViews of all relevant slots

  3. Apply targeting factor:
     totalRules = sum of rules across all draft.segments
     0 rules → factor 1.0
     1-2 rules → factor 0.6
     3-4 rules → factor 0.3
     5+ rules → factor 0.1

     ⚠ NOTE: This uses draft.segments (the legacy TargetingSegment[] field),
     NOT draft.audienceSegments (the newer AudienceSegment[] field). The current
     targeting UI (BrandTargetingSection, SellerTargetingSection) writes to
     draft.audienceSegments, which means the targeting factor is effectively
     always 1.0 in practice because draft.segments stays empty. This is a known
     inconsistency. If you want reach to respond to targeting changes, update
     this formula to count draft.audienceSegments instead.

  4. Apply frequency cap factor:
     If enabled: min(1, maxViews / 5)
     If disabled: 1.0

  5. Result = round(totalViews * targetingFactor * frequencyFactor)

16. BUDGET & SPEND CALCULATIONS (Managed CPM Flow)
================================================================================
(Expands on sections 7f and 12 above)

16a. Budget Summary (right side of schedule section)
─────────────────────────────────────────────────────
  campaignDays = ceil((endDate - startDate) / oneDay) + 1, or null if no end date
  isDaily = draft.budgetType === "daily"
  dailySpend = isDaily ? budget : (campaignDays ? budget / campaignDays : null)
  maxTotal = isDaily ? (campaignDays ? budget * campaignDays : null) : budget
  creditsAvailable = availableCredits.USD (hardcoded mock: $5,000)
  creditApplied = useCredits ? min(creditsAvailable, maxTotal || budget) : 0
  finalCost = (maxTotal || budget) - creditApplied

  Display rules:
  - Shows input budget (daily or total)
  - If campaignDays known: shows derived metric (daily→max total, total→daily)
  - If no end date: shows "Ongoing" note, hints to set end date for daily calc
  - Credits line: "Credits: -$X" with "(remaining: $Y)"
  - Final line: "Out-of-Pocket: $Z" or "$0 (Covered)" if fully credited

16b. Submit Review Section (CPM)
────────────────────────────────
  avgBid = average of audienceSegments bids, fallback to draft.bidAmount or $5
  slotMultiplierValue:
    - include mode with group multipliers: average of all group multiplier values
    - otherwise: single slotMultiplier value
  effectiveAvgBid = avgBid * (1 + slotMultiplierValue / 100)
  winRate = effectiveAvgBid < $3 → 30%, < $6 → 60%, else → 85%
  effectiveDailyBudget:
    - daily budget type: draft.budget directly
    - total budget type: draft.budget / campaignDays (or full budget if no duration)
  estDailyViews = round(min(estimatedReach, (effectiveDailyBudget / effectiveAvgBid) * 1000) * (winRate / 100))
  estTotalViews = estDailyViews * campaignDays (if known)
  estDailySpend = (estDailyViews * effectiveAvgBid) / 1000
  estTotalSpend = estDailySpend * campaignDays (if known)
  creditApplied = useCredits ? min(creditsAvail, estTotalSpend) : 0
  outOfPocket = max(0, estTotalSpend - creditApplied)

  Display: 4-column grid (Budget, Schedule, Est. Views, Win Rate)
  Then: "Estimated Spend Summary" bar with Est. Spend, Budget Cap,
  Avg Bid (eff.), Credits toggle, Out-of-Pocket

16c. Credits
────────────
  Mock credit balances (hardcoded in SinglePageFlow.tsx):
    USD: $5,000  |  AED: 18,500  |  SAR: 19,000  |  EGP: 155,000
  useCredits is a local boolean toggle in SinglePageFlow.tsx.
  Credits UI appears in both the budget summary and submit review.

17. SLOT MULTIPLIER SYSTEM (Managed Flow Only)
================================================================================
(Expands on section 7d above)

Slot multipliers increase the effective bid for specific ad zones.

Two modes:

  INCLUDE MODE (slotTargetingMode === "include"):
    Per-page-group multipliers. slotGroupMultipliers: Record<pageId, string>
    When user includes zones, they're grouped by page in the right panel.
    Each page group gets an input field for its multiplier %.
    Recommendation logic (inline "Rec: X%" text):
      >5M views/day → 5%
      >1M views/day → 10%
      >500K views/day → 15%
      Otherwise → 25%
    BiddingSection computes average of all group multipliers for display.

  EXCLUDE MODE (slotTargetingMode === "exclude"):
    Single global multiplier (slotMultiplier: string, default "0").
    Applies to all remaining (non-excluded) slots.

  ALL ZONES MODE:
    No multiplier UI shown.

Where multiplier is used:
  - BrandTargetingSection: input fields and display
  - BiddingSection: shows multiplier value, uses it in effective bid calc
  - SinglePageFlow submit review: uses averaged value for spend estimates

18. COMPOUND AUDIENCE SEGMENTS (BrandTargetingSection.tsx)
================================================================================
(Expands on section 7c above)

The managed flow allows ANDing multiple targeting conditions into one segment.

Data model:
  AudienceCondition: { type, name, description, lookback, category, subcategory, intent }
  AudienceSegment: { id, type, name, bid, suggestedBid, bidRange, conditions?: AudienceCondition[] }
  If conditions[] is present, it's compound. The top-level type/name are summaries.

UI flow:
  1. User browses three targeting buckets: Re-engagement, Demographic, In-Market
  2. Each bucket has dropdowns/options. Clicking "Add to Segment" adds an
     AudienceCondition to stagedConditions[] (local state).
  3. The "Segment Builder" area shows staged conditions as chips with "AND" labels.
  4. User enters a bid for the compound segment.
  5. Clicking "Add Segment →" calls commitStagedSegment() which creates a new
     AudienceSegment with the conditions[] array and adds it to draft.audienceSegments.
  6. The right panel shows all added segments. Compound segments display their
     sub-conditions as AND-joined chips.

Bid range for compound segments:
  getStagedBidRange() computes min/max across all staged conditions,
  then adds a +10% premium per additional condition beyond the first.

Self-serve (seller) flow does NOT have this. SellerTargetingSection adds
simple one-condition-per-segment entries directly.

24. IMPORTANT GOTCHAS & THINGS TO WATCH OUT FOR
================================================================================

1. SHALLOW MERGE: updateDraft() does a shallow merge ({ ...prev, ...updates }).
   For nested objects like builderConfig, you must spread the inner object too:
   updateDraft({ builderConfig: { ...draft.builderConfig, products: newProducts } })

2. LARGE FILES: SinglePageFlow.tsx is ~1850 lines. Section files are 400-920 lines.
   When editing, use precise search/replace. Read before editing.

3. DUPLICATE UI: The seller and brand flows in LandingPageSection.tsx have nearly
   identical JSX. Changes must be applied to BOTH. When using string replace,
   the same block may appear twice — use replace_all or add unique context.

4. REF TYPE MISMATCH: sectionRefs uses HTMLDivElement | null but section callbacks
   pass HTMLElement | null. This causes TS errors. Pre-existing, not critical.

5. PERFORMANCE: The original 7,175-line monolith caused lag because any state
   change re-rendered everything. The fix was React.memo on each section component.
   A useEffect was also introduced to replace a setTimeout inside render that was
   causing cascading re-renders in the daily CPT booking sync.

6. MOCK IMAGES: Product images use Unsplash URLs (may break if URLs expire).
   Template builder mock images are inline SVG data URIs (stable, self-contained).

7. ENTRY TYPE SIDE EFFECTS: Switching entryType in the header dropdown
   triggers side effects:
   seller → sets ownerType="self_serve", landingPageMode="builder", pricingModel="cpm"
   brand → sets ownerType="ops_managed" (other fields keep current values)

8. NO END DATE: Some CPM campaigns can run indefinitely (draft.noEndDate=true).
   Budget summary and submit review must handle null campaignDays gracefully.
   For no-end-date with daily budget, show "Ongoing" and skip total calculations.

9. SECTION VISIBILITY GATES:
   Seller: needs campaignName + startDate + budget>=100 to unlock later sections
   Brand CPM: needs startDate + budget>=100
   Brand CPT: needs slotIds.length>0 to show calendar, then cptTotalHours>0 for rest

29. BOOKINGS (formerly "Sales Intake")
================================================================================
(Rewritten March 6, 2026 — major model change)

CONCEPTUAL SHIFT:
  The old "Sales Intake" flow treated each intake as a single request that was
  "converted" into one campaign. The new model treats a Booking as a parent
  commercial/planning object that can result in MULTIPLE campaigns underneath it.
  This reflects the real-world reality where a single brand deal (e.g., Samsung
  Galaxy S25 Launch — $250K) spawns several campaigns across different pages,
  formats, and date ranges.

  Key changes from the old model:
  - "Sales Intake" renamed to "Bookings" throughout
  - "Convert to Campaign" removed — replaced with "Add Campaign"
  - BookingIntake.campaigns: BookingCampaign[] tracks child campaigns
  - Media Brief file upload removed — replaced with inline Media Plan table
  - AD_ASSET_TYPES / adAssetTypes fields removed from the model entirely
  - New statuses: in_planning, partially_converted, fully_converted
  - Old statuses removed: ready_for_ops, converted

29a. Navigation
  Top nav: Campaign Builder | Calendar Overview | Bookings
  Tab state: "builder" | "calendar" | "bookings" (page.tsx)
  Sub-navigation within the "bookings" tab:
    - List view (BookingsListPage): default when clicking the tab
